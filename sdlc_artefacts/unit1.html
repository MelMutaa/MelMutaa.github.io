Discussion on Cryptographic Failures and UML Representation

By Melissa Mutabazi - Published on August 28, 2024, 11:49 PM

OWASP's Top 10 provides a reference standard for identifying security risks in web applications and is an effective step toward producing secure code (OWASP, 2021). The OWASP Top 10 are released every four years and draw on data collected from projects to provide valuable insight about common vulnerabilities.

This discussion will focus on cryptographic failures. These failures are listed as the second most common vulnerability facing projects due to their vulnerability to exposing sensitive data or compromising systems (OWASP, 2021). Cryptographic failures can lead to the exposure of sensitive data (Kelly, 2023), such as personally identifiable information (PII) including financial information, health records, and other sensitive details.

OWASP classifies cryptographic failures as a symptom rather than a cause (OWASP, 2021). Any failure that exposes sensitive and critical data to an unauthorised party is considered a cryptographic failure (Hiremath, 2021). It is possible to present the design using a variety of UML models. For instance, a class diagram shows the structure of the system by displaying its classes, attributes, operations, and relationships. In this way, we can outline how data is handled by the application and how it interacts with the database. We can identify where user inputs are processed, and where SQL queries might be vulnerable.

Another UML model that can be used to present the design is a use case diagram to model the functional requirements of the system and the interactions between external users (actors) and the system. Use case diagrams help to identify the various scenarios where user inputs are accepted and processed. This makes it easier to pinpoint which use cases might be vulnerable to SQL Injection.
