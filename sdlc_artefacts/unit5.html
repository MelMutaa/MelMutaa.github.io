Should Cyclomatic Complexity be considered today in code design?

Cyclomatic Complexity (CC) remains valuable in understanding code complexity and maintainability, though it should not be the only metric used to gauge code quality. Its relevance is seen in a few key areas:

1. Testing and Maintainability: Cyclomatic Complexity helps determine how much testing is needed for software by counting the number of independent execution paths. High CC values suggest more testing is needed to ensure all possible paths are covered, improving code robustness (McCabe, 1976).

2. Maintainability: Code with high CC is often difficult to modify or extend without introducing errors. Lowering complexity through refactoring makes software easier to maintain, aligning with modern practices that emphasize clean, simple code (McCabe, 1976).

However, some argue that the importance of CC has decreased due to modern software development practices:

  • Unit Testing and Test-Driven Development (TDD): With modern testing frameworks, writing comprehensive test suites to cover all possible paths is more automated. Thus, CC becomes less crucial in determining test coverage (Prather, 2020).
  • Declarative and Functional Programming: In languages that emphasize declarative or functional styles (e.g., Haskell, Scala), CC might not adequately reflect actual code complexity since control flow is often implicit rather than explicit (Prather, 2020).

Is Cyclomatic Complexity relevant in developing secure software?

In the context of secure software development, Cyclomatic Complexity’s relevance is nuanced.

Arguments for Relevance:

1. Reducing Attack Surface: Higher code complexity often correlates with a greater likelihood of bugs or vulnerabilities, as complex code is harder to test and reason about. Reducing complexity through lower CC scores can help decrease the attack surface, making vulnerabilities easier to detect (SANS Institute, 2020).

2. Control Flow Complexity: Many security vulnerabilities, such as buffer overflows and input validation errors, often occur in highly complex control flows. Cyclomatic Complexity can act as an early indicator of areas that require closer scrutiny to ensure they are secure (Larson, 2018).

Arguments Against Relevance:

1. Security-Specific Issues: Cyclomatic Complexity does not address security-specific concerns such as data flow integrity, secure encryption, or proper handling of authentication and authorisation processes. Code with low CC could still have critical security flaws if these aspects are ignored (SANS Institute, 2020).

2. Modern Security Tools: Today, static analysis tools and automated vulnerability scanners are far more effective in identifying specific security risks compared to CC, which is more focused on control flow and logic (Weidman, n.d.).

Conclusion and Personal Opinion

Cyclomatic Complexity still plays a role in understanding and managing code complexity, which can indirectly support secure software development by highlighting areas of the code that require more attention during testing. However, in modern development, particularly when prioritizing security, CC is not as essential as other tools and methods specifically designed to identify vulnerabilities.

In my view, CC should be considered alongside other metrics, particularly in traditional procedural or object-oriented codebases where control flow is paramount. However, in environments where declarative or functional programming dominates, or where automated testing and security tools are heavily relied upon, its relevance is diminished. For secure software development, it is better to pair CC with security-focused practices such as static analysis, threat modelling, and secure coding standards.
